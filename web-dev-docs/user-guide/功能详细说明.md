# 功能详细说明

## 1. 文档生成功能

### 1.1 PRD生成

Web-Dev技能会根据用户输入自动生成产品需求文档（PRD），包含以下内容：

- **产品概述**：项目的简要描述、主要目的、目标用户和市场价值
- **核心功能**：用户角色（如有）、功能模块和页面详情
- **核心流程**：主要用户流程的自然语言描述和Mermaid流程图
- **用户界面设计**：设计风格、页面设计概述、响应式设计和3D场景指导（如适用）

### 1.2 技术架构文档生成

技术架构文档包含以下内容：

- **架构设计**：前端、后端（可选）、数据和外部服务的Mermaid diagram
- **技术描述**：前端技术栈、初始化工具、后端技术（如适用）和数据库选择
- **路由定义**：项目的路由结构和目的
- **API定义**：后端API的TypeScript类型定义、请求/响应模式（如适用）
- **服务器架构图**：控制器→服务→存储库→数据库的Mermaid diagram（如适用）
- **数据模型**：数据模型定义和数据定义语言（如适用）

### 1.3 文档存储

所有生成的文档都会存储在`.trae/documents/`目录中，方便用户查阅和修改。

## 2. 开发流程

### 2.1 设计思考

在开始编码之前，Web-Dev技能会帮助用户确定明确的美学方向：

- **目的**：界面解决什么问题？谁使用它？
- **风格**：选择一个极端的设计风格，如极简主义、 maximalist 混乱、复古未来主义等
- **约束**：技术要求（框架、性能、可访问性）
- **差异化**：什么让这个界面令人难忘？用户会记住的一件事是什么？

### 2.2 前端实现

Web-Dev技能会根据PRD和技术架构文档实现解决方案：

- **生产级代码**：编写功能完整、性能优化的代码
- **视觉吸引力**：创建视觉上引人注目、令人难忘的界面
- **设计一致性**：确保界面具有清晰的美学观点，保持一致性
- **细节完善**：精心完善每个细节，确保高质量

### 2.3 美学设计

Web-Dev技能遵循以下美学指南：

- **排版**：选择美观、独特的字体，避免通用字体
- **色彩与主题**：使用一致的色彩方案，使用CSS变量保持一致性
- **动效**：使用动画效果和微交互，提升用户体验
- **空间构图**：采用意外的布局、不对称设计、重叠元素等
- **背景与视觉细节**：创建氛围和深度，添加上下文效果和纹理

## 3. 技术栈支持

### 3.1 前端技术

- **React**：默认推荐React@18，用于构建用户界面
- **Tailwind CSS**：推荐Tailwind CSS@3，用于快速构建响应式界面
- **Vite**：推荐Vite作为构建工具，提供快速的开发体验
- **其他框架**：也支持Vue、Svelte等其他前端框架

### 3.2 3D支持

对于需要3D效果的项目，Web-Dev技能提供以下库的集成：

- **Three.js**：用于创建3D场景
- **@react-three/fiber**：React的Three.js包装器
- **@react-three/drei**：有用的Three.js助手组件
- **@react-three/postprocessing**：用于后期处理效果

### 3.3 工具集成

- **Vite**：用于项目初始化和构建
- **ESLint**：用于代码质量检查
- **Prettier**：用于代码格式化
- **Git**：用于版本控制

## 4. 响应式设计

Web-Dev技能默认采用桌面优先的设计方法，同时确保界面在不同设备上都能正常显示：

- **桌面设计**：为桌面设备优化布局和交互
- **移动适配**：确保在移动设备上有良好的显示效果
- **触摸优化**：为触摸设备优化交互元素

## 5. 语言一致性

Web-Dev技能严格遵循语言一致性规则：

- **输入语言**：检测用户输入的语言，并使用该语言生成所有输出
- **文档翻译**：将所有模板标题、描述和示例文本翻译成用户输入的语言
- **代码注释**：确保代码注释与用户输入的语言一致

## 6. 工作流程详细步骤

### 6.1 文档生成流程

1. **检查现有文档**：检查`.trae/documents/`目录中是否存在PRD和技术架构文档
2. **加载文档指南**：读取并遵循`web-docs-guideline.md`中的约束
3. **生成PRD**：根据用户需求生成产品需求文档
4. **生成技术架构文档**：生成详细的技术架构文档
5. **存储文档**：将所有文档存储在`.trae/documents/`目录中
6. **通知用户**：通知用户文档已准备就绪，需要批准

### 6.2 开发流程

1. **加载开发指南**：读取`web-dev-guideline.md`并严格遵循其中的指南
2. **实现解决方案**：根据PRD和技术架构文档实现功能
3. **应用设计指南**：应用美学指南，创建视觉吸引力强的界面
4. **测试功能**：确保所有功能正常工作
5. **优化性能**：优化代码性能，减少加载时间
6. **完成项目**：交付完整的Web项目

## 7. 功能限制

### 7.1 适用场景限制

- **绿色场域项目**：仅适用于从0到1的绿色场域Web开发项目
- **空工作区**：要求工作区为空或不包含前端代码
- **完整流程**：适用于遵循完整Web开发工作流程的项目

### 7.2 不适用场景

- **现有项目**：不适用于修改现有前端项目
- **Bug修复**：不适用于对现有代码进行bug修复
- **增量开发**：不适用于已建立代码库的增量开发

## 8. 技术约束

- **外部服务**：除非明确要求，否则尽量减少外部服务的使用
- **框架选择**：默认优先选择React，而非纯HTML
- **3D项目**：对于3D项目，使用Three.js及其相关库
- **设计方法**：默认采用桌面优先的设计方法

## 9. 最佳实践

### 9.1 文档创建最佳实践

- **详细描述**：提供详细的项目描述和功能需求
- **合理规划**：合理规划项目结构和技术栈
- **明确设计**：明确设计风格和视觉要求
- **定期更新**：根据项目进展定期更新文档

### 9.2 开发最佳实践

- **代码质量**：保持代码整洁、可维护
- **性能优化**：优化代码性能，减少加载时间
- **用户体验**：关注用户体验，确保界面直观、易用
- **响应式设计**：确保界面在不同设备上都能正常显示
- **安全考虑**：注意代码安全，避免常见的安全漏洞

### 9.3 设计最佳实践

- **风格一致**：保持设计风格的一致性
- **细节完善**：精心完善每个设计细节
- **创新思维**：尝试创新的设计方法和交互方式
- **用户中心**：以用户为中心，关注用户需求
- **可访问性**：确保界面对所有用户都可访问
